# Найденные баги

## Описание > Как тест поймал > Как исправили

### Баг №1: Отсутствует атрибут `season` в экземпляре класса

**Описание**  
В методе `__init__` класса `Echidna` параметр `season` принимается, но не сохраняется в `self`. При вызове `self.BMR()` внутри конструктора происходит обращение к `self.season`, которого нет, что вызывает `AttributeError`. Это блокирует создание любого объекта.

**Как тест поймал**  
Первый же позитивный тест (`test_01_default_creation`) пытается создать экземпляр `Echidna()`. В `setUp` каждого тестового класса вызывается `Echidna()`, и все 33 теста `unittest` упали с одинаковой ошибкой:  
`AttributeError: 'Echidna' object has no attribute 'season'`.  
Без исправления этого бага ни один другой тест не может быть выполнен.

**Как исправили**  
В `__init__` добавить строку:

```python
self.season = season
```

### Баг №2: Отсутствует валидация веса

**Описание**  
По спецификации вес ехидны должен быть в диапазоне 2.0–7.5 кг. При попытке создать ехидну с весом 1.9 кг или 8.0 кг должно выбрасываться `ValueError`. В текущей реализации валидации нет – объект создаётся с любым весом.

**Как тест поймал**  
Негативные тесты `test_01_weight_below_min` и `test_02_weight_above_max` ожидают `ValueError`, но фактически объект создаётся, и исключение не выбрасывается – тесты падают с `AssertionError: ValueError not raised`.

**Как исправили**  
В начале `__init__` добавить проверку:

```python
if not (self.MIN_WEIGHT <= weight <= self.MAX_WEIGHT):
    raise ValueError(f"Weight must be between {self.MIN_WEIGHT} and {self.MAX_WEIGHT}")
```

### Баг №3: Неверные названия температурных состояний

**Описание**  
Тест-план требует строгих имён состояний: `active`, `rem_sleep`, `hibernation`, `torpor`, `too_cold`, `overheat`. В коде используются другие имена: `overheating`, `fast_sleep`, `paralysis`, `critical`. Кроме того, для 25°C добавлен случайный фактор (50% шанс уйти в `fast_sleep`), что противоречит требованию.

**Как тест поймал**  
Позитивные тесты `test_05_rem_sleep_at_25`, `test_06_torpor_at_10`, `test_07_hibernation_at_33` и граничные тесты проверяют точное соответствие состояния. Ожидается `"rem_sleep"`, фактически приходит `"fast_sleep"` или `"active"` – тесты падают с `AssertionError`. Также тест `test_04_temp_25_boundary` проверяет, что при 25.0°C состояние `rem_sleep`, а при 25.1°C – `active`, но из-за случайности этот тест иногда проходит, иногда нет, что свидетельствует о недетерминированности.

**Как исправили**  

- Переименовать состояния в соответствии с тест-планом.

- Убрать случайность из `update_state()`, заменить на чёткие границы:  
  `>32` → `hibernation`, `25…32` → `active` (но 25°C ровно – `rem_sleep`), `5…15` → `torpor`, `<5` → `too_cold`, `>35` → `overheat`.  
- Реализовать `rem_sleep` как отдельное состояние ровно при 25°C.

### Баг №4: Ошибка в расчёте скорости рывка (sprint)

**Описание**  
В коде `SPRINT_SPEED = 24 / 1000 * 3600 = 86.4 км/ч`, тогда как по требованию скорость рывка – 24 км/ч. Из-за этого дистанция за 30 секунд получается не 0.2 км, а ~0.72 км.

**Как тест поймал**  
Тест `test_12_sprint_30sec` вычисляет ожидаемое расстояние `24 км/ч * 30/3600 ч = 0.2 км` и сравнивает с результатом `sprint(30/3600)`. Фактическое расстояние ~0.72 км, что не равно 0.2 – тест падает с `AssertionError`.

**Как исправили**  
Исправить формулу:  

```python
SPRINT_SPEED = 24   # км/ч
```

или  

```python
SPRINT_SPEED = 24 / 3.6   # м/с (если нужны м/с)
```

В методе `sprint` время передаётся в часах, поэтому оставить `24` как есть.

### Баг №5: Метод `shoot_tongue` не возвращает фиксированное количество выстрелов

**Описание**  
По требованию частота языка – 50 раз в минуту (ровно). В коде генерируется случайное число `normalvariate(50,10)`, причём метод ничего не возвращает, а только печатает. Тест-план ожидает проверяемое значение.

**Как тест поймал**  
Тест `test_14_tongue_frequency` пытается вызвать метод и проверить, что за 1 минуту было 50 выстрелов. Однако метод не возвращает число, поэтому тест может только проверить отсутствие исключений – что не является полноценной проверкой. Тест помечен как ожидающий падение (или просто проходит, но не проверяет суть). Это указывает на недостаточную тестируемость.

**Как исправили**  

- Убрать случайность, сделать частоту строго 50 выстрелов в минуту.

- Метод `shoot_tongue(minutes)` должен возвращать количество выстрелов: `return minutes * self.TONGUE_PER_MINUTE`.  
- При желании оставить случайность – вынести в отдельный режим, но по умолчанию – детерминированно.

### Баг №6: Отсутствует проверка энергии для языка

**Описание**  
Негативный тест «выстрел языка без энергии» требует выброса ошибки «нет энергии». В коде энергия не реализована, исключение не выбрасывается.

**Как тест поймал**  
Тест `test_05_tongue_no_energy` ожидает `Exception` (любую ошибку), но метод `shoot_tongue` выполняется без ошибок – тест падает с `AssertionError: Exception not raised`.

**Как исправили**  
Ввести атрибут `self.energy` (например, 100 единиц), а выстрел языка тратит 1 единицу. При `self.energy <= 0` выбрасывать `ValueError("no energy")`. После отдыха или еды энергию восстанавливать.
