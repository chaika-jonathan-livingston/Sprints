## Критические баги (блокирующие)

### БАГ №1 – Отсутствует атрибут `season` в экземпляре

**Описание**  
В конструкторе `Echidna` параметр `season` не присваивается `self`. При вызове `bmr()` (или `BMR()`) внутри `__init__` происходит обращение к `self.season`, которого нет, что вызывает `AttributeError`. Объект не может быть создан.

**Как тест поймал**  
Фикстура `default_echidna` пытается создать объект `Echidna()`. При первом же вызове фикстуры в любом тесте (например, `test_default_creation`) падает с ошибкой:  
`AttributeError: 'Echidna' object has no attribute 'season'`.  
Все тесты, использующие фикстуру, даже не стартуют – pytest показывает `ERROR` в сетапе.

```python
# Пример вывода pytest
test_echidna_pytest.py::test_default_creation ERROR
... AttributeError: 'Echidna' object has no attribute 'season'
```

**Как исправить**  
В `__init__` добавить:

```python
self.season = season
```

### БАГ №2 – Нет валидации веса

**Описание**  
Класс не проверяет, что вес находится в диапазоне [2.0, 7.5]. Можно создать ехидну с весом 1.9 или 8.0.

**Как тест поймал**  
Параметризованный тест `test_weight_boundaries` проверяет, что при весе 1.999 и 7.501 выбрасывается `ValueError`. Фактически объект создаётся без ошибок – тест падает с `Failed: DID NOT RAISE ValueError`.

**Как исправить**  
В начале `__init__` добавить:

```python
if not (self.MIN_WEIGHT <= weight <= self.MAX_WEIGHT):
    raise ValueError(f"Weight must be between {self.MIN_WEIGHT} and {self.MAX_WEIGHT}")
```

## Баги в логике состояний

### БАГ №3 – Неверные названия состояний

**Описание**  
По тест-плану ожидаются имена: `active`, `rem_sleep`, `hibernation`, `torpor`, `too_cold`, `overheat`. В коде используются: `overheating`, `fast_sleep`, `paralysis`, `critical`. Также для 25°C присутствует случайность (50% шанс уйти в `fast_sleep`), что не соответствует требованию.

**Как тест поймал:**

- Тест `test_rem_sleep_at_25` ожидает `"rem_sleep"`, получает `"fast_sleep"` или `"active"` – падение.  
- Тест `test_torpor_at_10` ожидает `"torpor"`, получает `"paralysis"` – падение.  
- Тест `test_hibernation_at_33` ожидает `"hibernation"`, получает `"overheating"` – падение.  
- Параметризованный тест `test_temperature_boundaries` проверяет точные границы (25.0 → rem_sleep, 25.1 → active). Из-за случайности этот тест иногда проходит, иногда нет – недетерминизм.

**Как исправить**  

- Переименовать состояния строго по таблице из тест-плана.  
- Убрать `random.random()` из `update_state()`.  
- Чётко задать:  
  - `temp == 25` → `"rem_sleep"`  
  - `25 < temp <= 32` → `"active"`  
  - `temp > 32` → `"hibernation"`  
  - `5 <= temp < 15` → `"torpor"`  
  - `temp < 5` → `"too_cold"`  
  - `temp > 35` → `"overheat"`

### БАГ №4 – Нет проверки энергии для языка

**Описание**  
Негативный тест требует, чтобы при нулевой энергии вызов `shoot_tongue()` бросал исключение. Механизм энергии не реализован.

**Как тест поймал**  
Тест `test_tongue_no_energy` ожидает любое исключение (`pytest.raises(Exception)`). Фактически метод выполняется без ошибок → тест падает с `Failed: DID NOT RAISE Exception`.

**Как исправить**  

- Ввести атрибут `self.energy` (например, 100).  
- При выстреле тратить 1 единицу.  
- При `self.energy <= 0` бросать `ValueError("no energy")`.  
- Метод `eat` должен восстанавливать энергию.

## Баги в расчётах и скоростях

### БАГ №5 – Ошибка в `SPRINT_SPEED`

**Описание**  
В коде `SPRINT_SPEED = 24 / 1000 * 3600 = 86.4 км/ч` (неправильно). Должно быть 24 км/ч.

**Как тест поймал**  
Тест `test_sprint_30sec` вычисляет ожидаемое расстояние: `24 км/ч * 30/3600 ч = 0.2 км`. Фактически получается `86.4 * 30/3600 ≈ 0.72 км`. `pytest.approx(0.2)` падает с `AssertionError`.

**Как исправить**  
Заменить на:

```python
SPRINT_SPEED = 24   # км/ч
```

### БАГ №6 – `shoot_tongue` не возвращает фиксированное число выстрелов

**Описание**  
По требованию частота – 50 раз/мин (строго). В коде генерируется случайное число, и метод ничего не возвращает.

**Как тест поймал**  
Тест `test_tongue_frequency` не может проверить точное число, так как метод не возвращает значение. Это делает невозможным полноценное тестирование – тест либо пропускается, либо проверяет только отсутствие исключений, что недостаточно. QA вынужден пометить тест как `xfail` или `skip`.

**Как исправить:**

1. Убрать случайность.
2. Метод `shoot_tongue(minutes)` должен возвращать `minutes * 50`.  
3. Дополнительно: добавить атрибут `tongue_shots` для подсчёта.

### БАГ №7 – В конструкторе `age` по умолчанию 15, а должно быть 0

Тест-план ожидает `age = 0` при создании. В коде `age = AVG_LIVE_LATENCY = 15`.
